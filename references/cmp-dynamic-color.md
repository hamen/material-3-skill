# CMP Dynamic Color

## Why This Matters

Dynamic color is the most visible Material Design 3 feature — it generates a full color scheme from the user's wallpaper, making the UI feel personal. But it's **Android 12+ only**. Every Compose Multiplatform project that targets iOS or Desktop must handle this gap. Without a proper fallback strategy, non-Android users get either a crash or no theme at all.

## Platform Availability

| Platform | Dynamic Color | Source | API |
|---|---|---|---|
| Android 12+ (API 31+) | Yes | System wallpaper | `dynamicLightColorScheme(context)` / `dynamicDarkColorScheme(context)` |
| Android < 12 | No | — | Falls back to seed-based scheme |
| iOS | No | No wallpaper extraction API | Must use seed-based scheme |
| Desktop (JVM) | No | No OS-level dynamic color | Must use seed-based scheme |

## The expect/actual Pattern

The standard CMP approach: define a common expectation, provide platform-specific implementations.

### Bridge Pattern Decision

Choose the right pattern for the right need:

| Need | Pattern | Rationale |
|---|---|---|
| ColorScheme per platform | `expect/actual` function | Stateless platform fact — no DI overhead needed |
| Theme with user preferences | Interface + DI (Koin) | Has state (user's chosen color), needs fakes for testing |
| Content-based dynamic color | Interface + DI | Async image extraction, testable with fake image source |

Dynamic color is a **stateless platform fact** — the OS either supports wallpaper extraction or it doesn't. That's why this file uses `expect/actual`, not interface+DI.

### commonMain

```kotlin
// commonMain/theme/PlatformColorScheme.kt
import androidx.compose.material3.ColorScheme
import androidx.compose.runtime.Composable

/**
 * Returns a platform-derived ColorScheme if available (e.g., Android dynamic color),
 * or null if the platform doesn't support dynamic color.
 */
@Composable
expect fun platformColorScheme(darkTheme: Boolean): ColorScheme?
```

### androidMain

```kotlin
// androidMain/theme/PlatformColorScheme.android.kt
// API-version-sensitive — requires Android 12 (API 31+) for dynamic color
import android.os.Build
import androidx.compose.material3.ColorScheme
import androidx.compose.material3.dynamicDarkColorScheme
import androidx.compose.material3.dynamicLightColorScheme
import androidx.compose.runtime.Composable
import androidx.compose.ui.platform.LocalContext

@Composable
actual fun platformColorScheme(darkTheme: Boolean): ColorScheme? {
    return if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.S) {
        val context = LocalContext.current
        if (darkTheme) dynamicDarkColorScheme(context) else dynamicLightColorScheme(context)
    } else {
        null // Fall back to seed-based scheme
    }
}
```

### iosMain

```kotlin
// iosMain/theme/PlatformColorScheme.ios.kt
import androidx.compose.material3.ColorScheme
import androidx.compose.runtime.Composable

@Composable
actual fun platformColorScheme(darkTheme: Boolean): ColorScheme? {
    return null // iOS has no wallpaper-based dynamic color
}
```

### desktopMain

```kotlin
// desktopMain/theme/PlatformColorScheme.desktop.kt
import androidx.compose.material3.ColorScheme
import androidx.compose.runtime.Composable

@Composable
actual fun platformColorScheme(darkTheme: Boolean): ColorScheme? {
    return null // Desktop has no OS-level dynamic color
}
```

## Complete Theme Setup

Full `AppTheme` composable that uses dynamic color when available and falls back gracefully:

```kotlin
// commonMain/theme/AppTheme.kt
import androidx.compose.foundation.isSystemInDarkTheme
import androidx.compose.material3.*
import androidx.compose.runtime.Composable
import androidx.compose.ui.graphics.Color

// Define your seed-based schemes (generated by Material Theme Builder or manually)
// Token mapping: md.sys.color.primary → Color(0xFF6750A4), etc.
private val LightColorScheme = lightColorScheme(
    primary = Color(0xFF6750A4),
    onPrimary = Color(0xFFFFFFFF),
    primaryContainer = Color(0xFFEADDFF),
    onPrimaryContainer = Color(0xFF21005D),
    secondary = Color(0xFF625B71),
    onSecondary = Color(0xFFFFFFFF),
    secondaryContainer = Color(0xFFE8DEF8),
    onSecondaryContainer = Color(0xFF1D192B),
    // ... add all roles or use generated Theme.kt from Material Theme Builder
)

private val DarkColorScheme = darkColorScheme(
    primary = Color(0xFFD0BCFF),
    onPrimary = Color(0xFF381E72),
    primaryContainer = Color(0xFF4F378B),
    onPrimaryContainer = Color(0xFFEADDFF),
    secondary = Color(0xFFCCC2DC),
    onSecondary = Color(0xFF332D41),
    secondaryContainer = Color(0xFF4A4458),
    onSecondaryContainer = Color(0xFFE8DEF8),
    // ... add all roles
)

@Composable
fun AppTheme(
    darkTheme: Boolean = isSystemInDarkTheme(),
    dynamicColor: Boolean = true,
    content: @Composable () -> Unit
) {
    val colorScheme = when {
        dynamicColor -> platformColorScheme(darkTheme) ?: if (darkTheme) DarkColorScheme else LightColorScheme
        darkTheme -> DarkColorScheme
        else -> LightColorScheme
    }

    MaterialTheme(
        colorScheme = colorScheme,
        typography = Typography(),
        shapes = Shapes(),
        content = content
    )
}
```

This pattern ensures:
- Android 12+: uses wallpaper-derived dynamic color (`md.sys.color.*` roles populated from wallpaper extraction)
- Android < 12: uses seed-based scheme
- iOS and Desktop: uses seed-based scheme
- The `dynamicColor` parameter allows users to disable dynamic color even on Android

## Fallback Strategies for Non-Android Platforms

When `platformColorScheme()` returns null, you need a fallback. Choose based on your app's needs:

### 1. Fixed Seed Color (Simplest)

Use [Material Theme Builder](https://material-foundation.github.io/material-theme-builder/) to generate `lightColorScheme` / `darkColorScheme` from your brand's seed color. Paste the generated Kotlin directly into `commonMain`.

**Best for:** Apps with a strong brand identity where the color scheme should be consistent everywhere.

### 2. User Preference / Theme Picker (Best UX)

Let users pick a seed color. Generate the scheme at runtime:

```kotlin
// Conceptual — API-version-sensitive
// The exact API depends on available color utilities in your CMP version.
// Use a color picker UI that saves the user's chosen seed color,
// then generate lightColorScheme / darkColorScheme from that seed.
// Libraries like MaterialKolor can generate M3 schemes from a seed
// color on all CMP targets.
```

**Best for:** Apps that want to give users personalization on all platforms, not just Android.

### 3. Brand Color Extraction (White-Label Apps)

Derive the seed color from the brand's primary color at build time or from a remote config:

```kotlin
// commonMain
// Map brand color to md.sys.color.primary and derive the rest
val brandSeedColor = Color(0xFF1A73E8) // From brand guidelines or remote config
// Use as the basis for lightColorScheme / darkColorScheme
```

**Best for:** White-label or multi-brand apps where the theme changes per deployment.

### 4. Content-Based Dynamic Color

Extract a dominant color from in-app content (album art, article images) and use it as the seed. This works on ALL platforms since it doesn't depend on the OS.

**Best for:** Media apps, e-commerce, any app where content drives the visual experience.

## Dark Theme Considerations Per Platform

| Platform | Dark Mode Detection | Dynamic Dark Scheme |
|---|---|---|
| Android | `isSystemInDarkTheme()` (reads system setting) | `dynamicDarkColorScheme(context)` on API 31+ |
| iOS | `isSystemInDarkTheme()` (reads iOS appearance) | Use seed-based `darkColorScheme()` |
| Desktop | `isSystemInDarkTheme()` (reads OS preference) | Use seed-based `darkColorScheme()` |

`isSystemInDarkTheme()` works cross-platform in Compose Multiplatform — it reads the correct OS setting on each platform. You do NOT need `expect/actual` for dark mode detection.

## Token Mapping

Dynamic color populates the same `md.sys.color.*` token roles as a seed-based scheme. The difference is only in the source of the seed:

| Token | Dynamic Color Source | Seed-Based Source |
|---|---|---|
| `md.sys.color.primary` | Derived from wallpaper | Derived from fixed seed |
| `md.sys.color.on-primary` | Derived from wallpaper | Derived from fixed seed |
| `md.sys.color.surface` | Derived from wallpaper | Derived from fixed seed |
| All other `md.sys.color.*` roles | Derived from wallpaper | Derived from fixed seed |

Components do not need to know whether dynamic color is active — they consume `MaterialTheme.colorScheme.*` and the correct values are always present.

## Testing Dynamic Color

### Android Emulator
1. Change the device wallpaper (Settings > Wallpaper)
2. Verify the app's color scheme updates to match
3. Test with both light and dark system themes
4. Test on API 30 (Android 11) to verify the seed-based fallback works

### iOS Simulator
1. Verify the seed-based color scheme renders correctly
2. Toggle dark mode (Settings > Developer > Dark Appearance) and verify dark scheme works
3. Confirm no crashes from missing dynamic color APIs

### Desktop
1. Run the desktop target
2. Toggle OS dark mode and verify the app responds
3. Confirm the seed-based scheme renders correctly
